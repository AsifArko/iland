# Vercel Environment Setup Guide

## Quick Setup with Interactive Script

The easiest way to set up your Vercel environment is using the built-in interactive script:

```bash
npm run vercel:setup
```

This script provides an interactive menu with the following options:

1. **Show setup instructions** - Step-by-step configuration guide
2. **List current environment variables** - View existing Vercel environment variables
3. **Pull environment variables** - Download variables to `.env.local`
4. **Validate current setup** - Check if all required variables are configured
5. **Show deployment workflow** - Understand the deployment process
6. **Exit** - Close the script

### Prerequisites

Before running the script, ensure you have:

- **Vercel CLI installed**: `npm i -g vercel`
- **Logged in to Vercel**: `vercel login`
- **Project linked**: The script will check your authentication and project setup

## Current Environment Files

### What You Currently Have:

1. **`.env`** - Base environment file (likely for Docker/production)
2. **`.env.local`** - Local development environment (used by Next.js)
3. **`env.example`** - Template file with all required variables (without dot prefix)

**Note**: Your `env.example` file is comprehensive and well-structured. You don't need a separate `.env.example` file with a dot prefix.

## Environment Strategy

### 1. Production Environment (Master Branch)

- **NODE_ENV**: `production` (auto-set by Vercel)
- **NEXT_PUBLIC_BASE_URL**: Your production domain
- **Stripe Keys**: Production keys
- **Sanity Dataset**: `production`
- **Email**: Production email settings

### 2. Preview Environment (Development Deployments)

- **NODE_ENV**: `production` (auto-set by Vercel)
- **NEXT_PUBLIC_BASE_URL**: Preview URL (auto-generated)
- **Stripe Keys**: Test keys (same as development)
- **Sanity Dataset**: `development` or `preview`
- **Email**: Test email settings

### 3. Local Development

- **NODE_ENV**: `development` (auto-set by Next.js)
- **NEXT_PUBLIC_BASE_URL**: `http://localhost:3000`
- **Stripe Keys**: Test keys
- **Sanity Dataset**: `development`
- **Email**: Test email settings

## Vercel Dashboard Configuration

### Step 1: Set Up Production Environment

1. Go to your Vercel project dashboard
2. Navigate to Settings → Environment Variables
3. Add all variables from your `.env.local` with production values

### Step 2: Set Up Preview Environment

1. In the same Environment Variables section
2. For each variable, click "Add" and select "Preview" environment
3. Use test/development values for sensitive services

### Step 3: Configure Branch Deployments

1. Go to Settings → Git
2. Configure branch deployments:
   - `master` → Production
   - `develop` → Preview
   - `feature/*` → Preview

### Production Environment Variables

```bash
# Stripe (Production)
STRIPE_SECRET_KEY=sk_live_...
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=pk_live_...
NEXT_PUBLIC_STRIPE_PRICE_ID=price_live_...

SMTP_PORT=587
SMTP_USER=your_production_email
```

## Code Adjustments Needed

### 1. Update Sanity Configuration

Your current Sanity config uses `production` dataset by default. For preview deployments, you might want to use a different dataset:

```typescript
// In sanity.config.ts
dataset: process.env.NEXT_PUBLIC_SANITY_DATASET || "production",
```

### 2. Handle Preview URLs

Your webhook and redirect URLs need to handle preview deployments:

```typescript
// In your webhook handlers

```

## Deployment Workflow

### For Development Deployments:

1. Push to `develop` branch → Creates preview deployment
2. Test with preview URL
3. Merge to `master` → Deploys to production

### For Feature Testing:

1. Create feature branch
2. Push changes → Creates preview deployment
3. Test with preview URL
4. Merge to `develop` or `master`

## Security Considerations

### 1. Environment Isolation

- Production and preview environments use different datasets
- Test Stripe keys for preview deployments
- Separate email accounts for testing

### 2. Webhook Configuration

- Configure Stripe webhooks for both production and preview URLs
- Use different webhook secrets for each environment

### 3. GitHub Token Security

- Use different GitHub tokens for different environments
- Limit token permissions appropriately

## Best Practices

1. **Never commit sensitive environment variables**
2. **Use different API keys for different environments**
3. **Test preview deployments before merging to production**
4. **Monitor both production and preview environments**
5. **Use descriptive branch names for better organization**

## Troubleshooting

### Common Issues:

1. **Environment variables not loading**: Check Vercel dashboard configuration
2. **Wrong dataset in preview**: Verify `NEXT_PUBLIC_SANITY_DATASET` setting
3. **Webhook failures**: Ensure webhook URLs are correctly configured
4. **Email delivery issues**: Check SMTP settings for each environment

### Debug Commands:

```bash
# Check environment variables in Vercel
vercel env ls

# Pull environment variables locally
vercel env pull .env.local

# Deploy with specific environment
vercel --prod  # Production
vercel         # Preview

# Use the interactive setup script
npm run vercel:setup
```
